## Commands

```bash
# Development (port 3001)
npm run dev

# Clear Next.js cache then start dev (use when hot reload misbehaves)
npm run devsafe

# Build & start production locally
npm run build && npm run start

# Lint
npm run lint

# Seed database with sample data (creates admin@campana.local / Campana2027!)
npm run seed

# After changing any Payload collection or global, regenerate types
npm run generate:types

# After adding custom admin components, regenerate the import map
npm run generate:importmap

# Database migrations (prefer over PAYLOAD_DB_PUSH in production)
npm run payload migrate:create   # create a new migration
npm run payload migrate          # apply pending migrations
```

### Tests

```bash
# Integration tests (Vitest + jsdom — no server needed)
npm run test:int

# Single integration test file
npx vitest run --config ./vitest.config.mts tests/int/api.int.spec.ts

# E2E tests (Playwright — requires running dev server)
npm run test:e2e
```

### Local setup

PostgreSQL runs in Docker on **port 5434** (non-standard to avoid conflicts with local installs). The dev server runs on **port 3001**.

```bash
cp .env.example .env
docker compose up -d db
npm install
npm run seed
npm run dev
```

Admin panel: `http://localhost:3001/admin`

## Architecture

### Two Next.js route groups

- `src/app/(frontend)/` — public-facing pages; all are React Server Components that call Payload directly
- `src/app/(payload)/` — Payload CMS admin panel and REST/GraphQL API routes (auto-generated; rarely edited manually)

### Payload embedded in Next.js

Payload CMS runs inside the same Next.js process. There is no separate CMS server. Every frontend page imports and calls `getClient()` from `src/lib/payload.ts`, which returns a cached Payload instance for direct in-process queries — no HTTP round-trip.

```ts
// Typical page pattern
const payload = await getClient()
const docs = await payload.find({ collection: 'noticias', ... })
```

### Collections and Globals

Collections (documents with many records): `Propuestas`, `Noticias`, `Entrevistas`, `Contactos`, `Sugerencias`, `TestigosElectorales`, `Reuniones`, `Media`, `Users` — all defined in `src/collections/`.

Globals (single-record config): `HojaDeVida` (candidate bio) and `AjustesSitio` (site name, logo, social links, footer text) — defined in `src/globals/`. These are fetched on nearly every page via `payload.findGlobal()`.

### Forms use Server Actions

Each participation form has a co-located `actions.ts` with a Next.js Server Action that writes the submission to the corresponding Payload collection (`Contactos`, `Sugerencias`, `TestigosElectorales`, `Reuniones`).

### Slug generation

`src/fields/slug.ts` exports `slugField()` (a reusable Payload field) and `slugify()` (a pure function). Slug is auto-derived from the `titulo` field on `beforeValidate` if left empty. Used by `Propuestas` and `Noticias`.

### Brand/styling

Colors are defined as CSS custom properties in `src/app/(frontend)/globals.css` under `@theme` (Tailwind v4 syntax). Use `var(--color-marca)` etc. in components rather than hardcoding hex values. The palette is MIRA party branding (blue `#003e8e`, gold `#f8b018`).

Rich text from Payload (Lexical editor) is rendered via `src/components/RichText.tsx` and styled with the `.prosa` CSS class.

### TypeScript types

`src/payload-types.ts` is auto-generated by Payload from the collection/global definitions. Never edit it manually; regenerate with `npm run generate:types` after any schema change.

### Database

PostgreSQL via `@payloadcms/db-postgres`. In development, Payload auto-pushes schema changes. For production, use migrations (`npm run payload migrate:create` / `migrate`). The `PAYLOAD_DB_PUSH=true` env var forces a push on first production deploy.
